# DECISIONS.md

## 1. 문서 목적

이 문서는 기능 목록을 다시 설명하는 문서가 아니라, 이 시스템을 설계하면서 어떤 문제를 중요하게 보았고 어떤 선택을 했는지 정리하기 위한 문서입니다.

이번 설계에서 가장 중요하게 본 것은 아래와 같습니다.

- 초과 판매를 끝까지 막는 것
- Redis 장애 시에도 예약을 완전히 멈추지 않는 것
- 깨진 정합성을 복구 가능한 구조로 만드는 것
- 중복 요청과 결제 실패까지 설명 가능한 상태로 남기는 것

---

## 2. 재고 정합성의 기준

이 시스템에서 가장 중요한 불변조건은 **최종적으로 성공한 예약 수가 총 재고를 초과하지 않는 것**입니다.

저는 이 문제를 단순히 재고 숫자를 빠르게 줄이는 문제로 보지 않았습니다.  
오히려 **재고 1개가 누구에게 점유되었는지를 어떤 상태로 증명할 수 있는가**가 더 중요하다고 보았습니다.

Redis 장애 시에는 결국 RDB를 기준으로 재고를 판단해야 하므로, 재고 공식은 아래처럼 정의했습니다.

```text
availableStock = totalStock - activeBookingCount
activeBookingCount = PENDING_PAYMENT + CONFIRMED
```

이 공식이 성립하려면 아래 조건이 반드시 지켜져야 합니다.

- 재고 확보 성공 이후에만 `PENDING_PAYMENT` booking이 생성되어야 합니다.
- 즉 `Booking`은 단순 주문 데이터가 아니라, 이미 재고 1개가 점유되었다는 사실의 근거여야 합니다.

### 트레이드오프

- `PENDING_PAYMENT`도 재고 점유로 보기 때문에 일시적인 미달 판매는 생길 수 있습니다.
- 대신 초과 판매를 막고, 장애 시에도 같은 재고 공식을 유지할 수 있습니다.

---

## 3. 재고 점유 시점

재고는 `GET Checkout`에서 선점하지 않고, `POST Booking`에서만 점유하도록 설계했습니다.

### 이유

- Checkout 진입 시점에 재고를 잡으면 실제 결제로 이어지지 않는 요청 때문에 재고가 오래 묶일 수 있습니다.
- 이 경우 초과 판매는 막더라도 실제로는 팔 수 있었던 재고가 묶이는 미달 판매가 생길 수 있습니다.
- 그래서 실제 구매 의사가 확인된 `POST Booking`만 재고 점유 지점으로 두었습니다.

### 트레이드오프

- Checkout에 진입한 사용자가 실제 Booking 단계에서 품절을 경험할 수 있습니다.
- 대신 불필요한 선점 없이 실제 판매 가능성을 더 높일 수 있습니다.

---

## 4. 재고 저장소 역할 분리

정상 경로의 실시간 재고 할당은 Redis가 담당하고, 예약/결제의 영속 기록과 장애 복구 기준은 RDB가 담당하도록 역할을 나누었습니다.

### 역할 분리

- Redis: 정상 경로의 빠른 재고 할당 엔진
- RDB: 예약/결제의 영속 기록 저장소
- RDB: fallback과 recovery의 기준 데이터

### 이유

- 00시 오픈 시점의 경쟁 요청을 모두 RDB로 처리하면 잠금 경합과 I/O 부하가 커집니다.
- 반면 Redis만 절대적인 진실로 두면 장애 시 복구 기준이 약해집니다.
- 그래서 실시간 할당은 Redis, 다시 믿고 복구할 기준은 RDB로 분리했습니다.

### 트레이드오프

- 구조는 단일 저장소보다 복잡해집니다.
- 대신 성능과 복구 가능성을 함께 가져갈 수 있습니다.

---

## 5. 선착순 처리와 동시성 제어

정상 경로의 재고 차감은 Redis Lua Script로 처리했습니다.

### 이유

- `조회 -> 차감`을 여러 명령으로 나누면 race condition이 생길 수 있습니다.
- Lua Script를 사용하면 아래를 하나의 원자적 연산으로 묶을 수 있습니다.
  - 재고 확인
  - 중복 요청 확인
  - 재고 차감
  - holder 기록

### 제가 정의한 선착순

이 시스템에서 선착순은 사용자 클릭 순서가 아니라, **재고 할당기에 먼저 도달해 재고 확보에 성공한 요청 순서**로 보았습니다.

또한 시간 기준은 서버 로컬 시간보다 **현재 경로에서 가장 신뢰할 수 있는 권위 시간**을 우선하도록 판단했습니다.

- 정상 경로: Redis 기준 시간
- fallback 경로: DB 기준 시간

### 트레이드오프

- 절대적인 클릭 순서 공정성을 보장하는 것은 아닙니다.
- 대신 시스템 내부에서는 일관된 기준으로 설명 가능한 공정성을 유지할 수 있습니다.

## 6. 장애 전환과 정합성 복구

Redis 장애가 발생했을 때 전체 요청을 차단하는 방식도 가능했지만, 저는 그 방식을 선택하지 않았습니다.

### 이유

- 장애 때문에 먼저 온 사용자도 기회를 잃게 만들고 싶지 않았습니다.
- 성능보다 정합성과 서비스 지속성이 더 중요하다고 보았습니다.
- 그래서 Redis가 비정상이면 `DB_FALLBACK`으로 전환해 RDB 기준으로 계속 예약을 받도록 했습니다.

운영 모드는 아래 3가지를 둡니다.

- `REDIS_NORMAL`
- `DB_FALLBACK`
- `RECOVERING`

### 제가 가장 위험하게 본 구간

제가 가장 무섭게 본 구간은 아래였습니다.

1. Redis에서는 재고 확보 성공
2. 그 직후 Booking 생성 시도
3. 이 시점에 RDB 또는 서버 장애 발생
4. Redis에는 차감 흔적이 남지만 RDB에는 booking이 없음

이 상태를 그대로 두면 fallback 재고 공식이 깨집니다.  
그래서 단순 예외 처리만으로는 부족하다고 보았고, **즉시 보상 + recovery worker** 두 단계가 모두 필요하다고 판단했습니다.

### 복구 방식

- 문제 A: Redis 재고 확보 후 Booking 미생성
- 문제 B: 오래된 `PENDING_PAYMENT`
- `RECOVERING`: DB 기준으로 Redis 재동기화 후 정상 복귀

### 트레이드오프

- 구조는 더 복잡해집니다.
- 대신 장애가 나도 정합성을 다시 맞출 수 있습니다.

---

## 7. Redis holder 관리

`remaining`만으로는 부족하다고 판단했습니다.

- 재고가 왜 줄었는지 알 수 없습니다.
- 어떤 요청이 점유했는지 알 수 없습니다.
- booking 생성 전 중복 요청을 막기 어렵습니다.
- orphan allocation 복구 근거가 없습니다.

그래서 Redis에는 아래 두 값을 함께 둡니다.

- `stock:product:{productId}:remaining`
- `stock:product:{productId}:holders`

### holders의 역할

- booking 생성 전 중복 요청 방지
- Redis 차감 후 booking 미생성 상태 복구 근거
- 결제 실패 시 중복 반납 방지
- `RECOVERING` 시 Redis 재구성 근거

### 한 줄 정리

`holders`는 단순 보조 키가 아니라, **Redis 재고 차감과 RDB booking 생성 사이의 공백을 설명 가능하게 만드는 임시 원장**입니다.

---

## 8. 멱등성 처리

멱등성은 `checkoutTokenId` 기준으로 처리했습니다.

### 역할 분리

- DB: 이미 생성된 요청의 최종 진실
- Redis `holders`: booking 생성 전 중복 차감 방지

즉 중복은 한 군데에서만 막지 않았습니다.

- 생성 후 중복: DB unique + 기존 결과 재반환
- 생성 전 중복: Redis holders

### 이유

선착순 시스템에서는 “이미 생성된 요청”과 “아직 생성 전 처리 중인 요청”의 중복이 다르기 때문입니다.

---

## 9. 결제 실패와 보상 처리

결제 실패는 단순 에러 응답이 아니라, booking / payment / point / stock을 함께 정리해야 하는 상태 전이 문제로 보았습니다.

### 실패 시 처리 순서

- `Payment FAILED`
- `PointHold RELEASED`
- `Booking FAILED`
- Redis 정상 경로였다면 재고 release

### 이유

- 내부 상태를 먼저 실패로 확정해야 현재 요청이 어떤 상태인지 해석 가능합니다.
- 그 뒤 point/stock을 복구해야 recovery와 운영 관점에서도 일관성이 유지됩니다.

---

## 10. 결제 확장성

결제에서 가장 중요하게 본 것은 아래 두 가지였습니다.

- 새 결제수단이 추가돼도 Booking 로직 수정이 최소여야 합니다.
- 결제 금액 조합 검증은 서비스가 아니라 도메인이 가져야 합니다.

그래서 아래 구조를 선택했습니다.

- `PaymentService`
- `PaymentProcessor`
- `PaymentStrategyFactory`
- `PaymentStrategy`
- `Payment`

### 역할 분리

- Booking: 예약 흐름 조율
- Payment: 결제 실행과 해석
- Payment 도메인: 금액 계산과 조합 검증

### 결과

- `POINT_ONLY`
- `CARD_ONLY`
- `YPAY_ONLY`
- `CARD_WITH_POINT`
- `YPAY_WITH_POINT`

을 도메인 규칙으로 다룰 수 있고, 새 결제수단 추가 시 Booking 비즈니스 로직 수정 범위를 줄일 수 있습니다.

---

## 11. 포인트 점유와 복합 결제

포인트는 바로 차감하지 않고 먼저 `HELD` 상태로 잠그는 방식을 선택했습니다.

### 이유

- `TX-1`과 `TX-2` 사이에 외부 결제와 장애 구간이 있습니다.
- 이 시점에 포인트를 바로 차감하면 실패 시 보상이 복잡해집니다.

그래서 포인트는 아래처럼 처리합니다.

- 시작 시: `HELD`
- 성공 시: `COMMITTED`
- 실패 시: `RELEASED`

### 의미

`PointHold`는 단순 임시 데이터가 아니라, **이 포인트는 결제 결과가 확정될 때까지 다른 요청이 쓰면 안 된다는 점유 상태**입니다.

### 트레이드오프

- 상태 전이가 늘어나 구현은 복잡해집니다.
- 대신 포인트 장부 정합성과 실패 보상이 단순해집니다.
